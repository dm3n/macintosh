# Operator Workflows

This page explains how Macintosh is meant to be used day-to-day as one coherent product.

## Workflow A: Local Product Development

1. Start in Superset for implementation work.
2. Use Codex/Claude/Gemini/OpenCode based on task fit.
3. Pull context from PKB before major architectural decisions.
4. Implement, verify, and document in the relevant repository.

Outcome: fast local iteration with persistent context.

## Workflow B: Remote AI/Security Operations

1. Open terminal (Superset or Warp).
2. Run `ssh kali`.
3. Perform Linux-side AI repository or cybersecurity tasks.
4. Exit session when finished.

Outcome: frictionless remote operations from any location.

## Workflow C: Approval-Safe Automation Delivery

1. Agent Hub domain agents propose pending actions.
2. Approval gateway syncs actions to Linear.
3. Human approves/rejects.
4. Executor delivers only approved actions.
5. Logs/audit state are persisted.

Outcome: controlled automation with explicit human gate.

## Workflow D: Knowledge Compounding Loop

1. Capture raw inputs to Brain/Raw or session logs.
2. Process and compile into structured Wiki/Memory pages.
3. Query PKB during future build or strategy work.
4. Feed new validated outcomes back into memory.

Outcome: every session increases system intelligence over time.

## Unified Narrative

Macintosh is intentionally split but operationally unified:
- local Mac layer for creation and control
- personal cloud layer for durable runtime execution
- Kali node for Linux AI/security depth
- approval and memory loops for safety and compounding

This creates a system that is fast to use, safe to operate, and coherent to scale.
